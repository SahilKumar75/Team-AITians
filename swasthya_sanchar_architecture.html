Decentralised Medical Records System Architecture & Code Flows

Patient-sovereign, blockchain-anchored, zero third-party auth

1. Structure & Project Architecture

The system follows an MVVM (Model-View-ViewModel) inspired architecture on the frontend, with Web3 smart contracts acting as the distributed backend state, and IPFS as the decentralized storage layer.

Folder Structure

  • /src/app: Next.js 14 App Router pages (Views). Divided into patient, doctor, hospital, and emergency routes.
  • /src/lib: Core logic, cryptography (crypto-identity.ts, crypto-files.ts), Helia integrations, local RxDB logic.
  • /contracts: Solidity smart contracts (Model) for Polygon Amoy blockchain.
  • /docs: Architectural references, data storage, and role documentation.

Core Concepts

  • Decentralized No central databases. Data lives on IPFS & Blockchain.
  • Sovereign Auth "Zero MetaMask". Wallet generated implicitly via email/password.
  • Local First RxDB / IndexedDB used for fast, offline capabilities.

2. Technology Stack & Selection Rationale

Technology Layer Why Selected?
Next.js 14 (App Router) Frontend Allows robust routing, API endpoints, and fast React-based UI development.
Polygon Amoy Testnet Blockchain Fast transaction speeds, exceptionally low gas fees, Ethereum-compatible EVM.
Helia (self-hosted) IPFS Node Replaces third-party pinning (like Pinata) to ensure true decentralization within the app.
RxDB (IndexedDB) Local DB Provides instant load states and offline-first capabilities for mobile users.
Brotli WASM Compression Provides 40-60% smaller IPFS payloads, crucial for saving expensive bandwidth on medical imaging (X-Rays).
Web Crypto API + noble/hashes Cryptography Ensures robust client-side hashing and encryption (AES-256-GCM) without third-party authentication services.
Fleek.co Hosting Deploys directly to IPFS, removing vendor lock-in.

3. Smart Contract Information

IdentityRegistry.sol

Manages user identities (Patient, Doctor, Hospital) matching a hashed identifier with user's encrypted wallets.

  • Struct Identity: Stores lockACid, lockCCid (legacy placeholder), recoveryKeyHash (legacy placeholder), emergencyCid (Tier 0 public profile), wallet address, role, and license hash.
  • Capabilities: Adding/removing guardians, social recovery mechanics, lock updates. Note: Recovery phrases have been superseded by Social Recovery.
  • Role Verification: Doctor verification uses a decentralized IVerifier interface before granting privileges.

HealthRegistry.sol

Manages references to IPFS records, per-document access lists, and emergency protocols.

  • Optimization: Encrypted DEKs are not stored in contract state arrays (costs $$$). Instead, they are fired as AccessGranted(encDekIpfsCid) events, reducing gas fees by 99.6%.
  • Verification Gate: require(verifier.isVerified(grantee)) ensures only real doctors get access.
